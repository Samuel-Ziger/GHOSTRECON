ESCOPO DO PROJETO

Aplicação web: https://foconopapiro.com.br — plataforma de estudos com sistema de ranking, metas, progresso e planos de assinatura (gratuito e premium).

Backend: Supabase (projeto suwzslcvitjudebflstl.supabase.co) — banco de dados PostgreSQL gerenciado com API REST autogenerada, autenticação via GoTrue (/auth/v1/) e políticas de Row Level Security (RLS).

Endpoints de autenticação: POST /auth/v1/token?grant_type=password — endpoint de login por e-mail e senha, sem limitação de tentativas.

Endpoints de dados: /rest/v1/study_records (registros de sessões de estudo), /rest/v1/user_plans (planos e status de assinatura premium, campos Stripe).

Contexto de lógica de negócio: sistema de assinatura integrado ao Stripe para controle de plano premium; dados de estudo utilizados diretamente para ranking, progresso e metas.

VULNERABILIDADE

1. [NÃO CORRIGIDO] Ausência de Rate Limiting e de Autenticação de Dois Fatores no Endpoint de Login

SEVERIDADE

MÉDIA

DESCRIÇÃO

Foi identificado que o endpoint de autenticação POST /auth/v1/token?grant_type=password, exposto pelo Supabase da aplicação foconopapiro.com.br, não implementa qualquer mecanismo de limitação de tentativas (rate limiting) nem exige autenticação de dois fatores (2FA/MFA).

Isso significa que um atacante que conheça o endereço de e-mail de um usuário pode realizar um ataque de força bruta no campo password de forma irrestrita, iterando wordlists (ex.: rockyou) ou combinações comuns sem ser bloqueado, sem espera progressiva (backoff) e sem notificação ao usuário legítimo.

Adicionalmente, a ausência de 2FA garante que, uma vez obtida a senha, o acesso à conta é imediato e completo, sem nenhuma camada adicional de verificação de identidade.



CENÁRIO DE ATAQUE

Um atacante que possua ou descubra o endereço de e-mail de um usuário da plataforma pode:

Iterar automaticamente listas de senhas comuns contra o endpoint de login

Obter acesso à conta sem qualquer bloqueio ou alerta

Acessar dados de estudo, progresso, ranking e status de assinatura da vítima

Dependendo do nível de privilégio da conta comprometida, potencialmente encadear com outras vulnerabilidades documentadas neste relatório



RECOMENDAÇÃO

Para mitigar essa vulnerabilidade, recomenda-se:

Implementar rate limiting no endpoint de autenticação (ex.: máximo de 5 a 10 tentativas por IP/e-mail em janela de tempo com backoff progressivo)

Habilitar e promover o uso de autenticação de dois fatores (TOTP/MFA) para todos os usuários, com obrigatoriedade para contas com privilégios elevados

Implementar bloqueio temporário ou CAPTCHA após número definido de tentativas falhas

Monitorar e alertar sobre tentativas consecutivas de login com falha (SIEM/logs)

[NÃO CORRIGIDO] Business Logic Flaw — Backend sem Validação de Input em Registros de Estudo (Injeção de Dados Falsos)

SEVERIDADE

ALTA

DESCRIÇÃO

Foi identificada uma falha de lógica de negócio no endpoint POST /rest/v1/study_records da aplicação foconopapiro.com.br. O backend aceita e persiste registros de estudo com valores completamente irreais sem qualquer validação de consistência ou plausibilidade.

Durante os testes, foi possível inserir registros com:

duration_seconds: 2147483647 (valor máximo de inteiro de 32 bits, equivalente a aproximadamente 596.523 horas ou 68 anos de estudo contínuo)

questions_count: 100 e correct_count: 100 (100% de acerto com quantidade arbitrária)

subject: qualquer valor arbitrário (ex.: "PoC_MAX")

record_date: qualquer data (incluindo datas futuras ou retroativas)

O sistema aceitou esses registros como válidos e os incorporou diretamente às métricas de ranking, progresso, metas e analytics da plataforma, sem qualquer mecanismo de detecção de fraude.



CENÁRIO DE ATAQUE

Um usuário autenticado mal-intencionado pode:

Inserir registros de estudo com valores absurdos para dominar rankings artificialmente

Falsificar progresso e metas da plataforma sem estudo real

Manipular dados de analytics e dashboards utilizados pela organização

Prejudicar usuários legítimos cujas posições de ranking são afetadas pela manipulação

Explorar quaisquer benefícios ou recompensas vinculados ao desempenho na plataforma

No cenário validado, a combinação de cegueira do backend + uso direto dos dados em funcionalidades críticas representa comprometimento total de integridade dos dados da plataforma.



RECOMENDAÇÃO

Para mitigar a vulnerabilidade, recomenda-se:

Implementar validação server-side de todos os campos: limites máximos plausíveis para duration_seconds (ex.: máximo de 8-12 horas por sessão), questions_count, correct_count (não pode exceder questions_count), etc.

Validar consistency entre campos: correct_count ≤ questions_count, duration_seconds > 0 e dentro de faixa razoável

Implementar limite diário de horas registradas por usuário

Nunca confiar nos dados enviados pelo cliente para cálculos críticos de ranking e métricas — recalcular no servidor com base em dados confiáveis

Implementar monitoramento e alertas para valores estatisticamente anômalos

Auditar e recalcular os rankings e dados históricos comprometidos

[NÃO CORRIGIDO] Broken Access Control — Exposição de Dados de Assinatura via API Supabase (user_plans)

SEVERIDADE

ALTA

DESCRIÇÃO

Foi identificado que a tabela user_plans do Supabase da aplicação foconopapiro.com.br está acessível via API REST por qualquer usuário autenticado, expondo dados sensíveis de assinatura, incluindo:

plan_type (tipo de plano: free ou premium)

premium_expires_at (data de expiração da assinatura)

A política de Row Level Security (RLS) do Supabase não está configurada de forma adequada para restringir o acesso de leitura exclusivamente aos dados do próprio usuário, permitindo que qualquer token autenticado consulte e enumere registros de planos.

Um usuário autenticado mal-intencionado pode:

Enumerar e listar dados de assinatura de outros usuários da plataforma

Obter identificadores Stripe (customer_id, subscription_id) de terceiros

Mapear quais usuários possuem plano premium e suas datas de expiração

Utilizar as informações coletadas para direcionar ataques adicionais (engenharia social, ataques ao Stripe, etc.)



RECOMENDAÇÃO

Implementar e revisar políticas de RLS no Supabase para a tabela user_plans, garantindo que cada usuário só possa acessar seu próprio registro:

CREATE POLICY "Usuário acessa apenas seus dados"
ON user_plans
FOR SELECT
USING (auth.uid() = user_id);

Verificar e revisar todas as políticas RLS de todas as tabelas sensíveis

Nunca expor identificadores de sistemas externos (Stripe IDs) desnecessariamente via API pública

Realizar auditoria completa das permissões de todas as tabelas do projeto Supabase

Um atacante autenticado pode:

Corromper o estado de assinatura de sua própria conta, causando inconsistência entre banco e Stripe

WITH CHECK (plan_type = OLD.plan_type AND premium_expires_at = OLD.premium_expires_at);

Bloquear alterações diretas em todos os campos relacionados ao Stripe e ao status de pagamento

[NÃO CORRIGIDO] Escalada de Privilégio de Pagamento — Ativação de Plano Premium sem Pagamento via PATCH Direto na API

SEVERIDADE

CRÍTICA

DESCRIÇÃO

Foi identificada uma vulnerabilidade crítica de lógica de negócio na aplicação foconopapiro.com.br: qualquer usuário autenticado pode ativar o plano premium da plataforma sem realizar qualquer pagamento, simplesmente enviando uma requisição PATCH direta ao endpoint /rest/v1/user_plans do Supabase com os campos plan_type e premium_expires_at.

Durante os testes, foi possível:

Alterar plan_type de "free" para "premium" sem validação de pagamento

Definir premium_expires_at para "9999-12-31T23:59:59Z" (expiração praticamente nula)

Obter acesso imediato a todas as funcionalidades premium da plataforma sem cobrança

Manter o status ativo indefinidamente sem nenhuma verificação posterior do Stripe

A falha ocorre porque a política de Row Level Security (RLS) do Supabase não impede que o usuário autenticado modifique campos críticos de seu próprio registro na tabela user_plans, e o sistema não valida o estado de pagamento com o Stripe antes de conceder acesso premium.



CENÁRIO DE ATAQUE

Um atacante autenticado com uma conta gratuita pode:

Enviar uma única requisição PATCH ao endpoint de assinatura

Obter acesso premium imediato e permanente sem pagamento

Contornar todo o sistema de cobrança e monetização da plataforma

Compartilhar o método com outros usuários, comprometendo as receitas da organização

Escalar o ataque para contas de outros usuários (em combinação com IDOR documentado)

O impacto financeiro é direto e pode ser explorado por qualquer usuário cadastrado, sem necessidade de conhecimento técnico avançado.



RECOMENDAÇÃO

Implementar política RLS restritiva que bloqueie qualquer alteração direta aos campos de plano:

CREATE POLICY "Bloquear alteração de plano pelo usuário"
ON user_plans
FOR UPDATE
USING (auth.uid() = user_id)
WITH CHECK (
  plan_type = (SELECT plan_type FROM user_plans WHERE user_id = auth.uid())
  AND premium_expires_at = (SELECT premium_expires_at FROM user_plans WHERE user_id = auth.uid())
);

Gerenciar mudanças de plano exclusivamente via função de servidor segura (Edge Function ou backend próprio) com validação de webhook do Stripe

Nunca confiar em campos de assinatura enviados diretamente pelo cliente

Implementar auditoria e alertas para mudanças na tabela user_plans

Revisar e recalcular contas com premium_expires_at suspeito (ex.: datas distantes como 9999)

Tratar como incidente: identificar e reverter contas que ativaram premium indevidamente

CRÍTICA



CONCLUSÃO

No período de 28/04/2026 a 30/04/2026, a avaliação sobre o alvo https://foconopapiro.com.br no âmbito do projeto foconopapiro confirmou uma superfície de risco relevante, centrada no backend Supabase exposto publicamente e na ausência de controles adequados de validação e autorização.

Os achados validados e documentados no corpo deste relatório incluem: (i) ausência de rate limiting e de autenticação de dois fatores no endpoint de login (/auth/v1/token), permitindo ataques de força bruta irrestrita; (ii) falha de lógica de negócio no endpoint de registros de estudo (/rest/v1/study_records), que aceita valores completamente irreais (ex.: 596.523 horas de estudo) sem qualquer validação server-side, comprometendo ranking, progresso e analytics; (iii) broken access control na tabela user_plans, com exposição de dados sensíveis de assinatura e identificadores do Stripe a qualquer usuário autenticado; (iv) manipulação direta de campos do Stripe (stripe_subscription_id, stripe_customer_id, plan_type, premium_expires_at) via PATCH sem validação com o provedor de pagamento, gerando estados inconsistentes; e (v) escalada de privilégio de pagamento crítica, que permite a qualquer usuário autenticado ativar o plano premium da plataforma sem pagamento, com expiração configurável para o ano 9999, através de uma única requisição PATCH ao endpoint de assinatura.

Em conjunto, estes problemas permitem a qualquer usuário cadastrado na plataforma contornar completamente o sistema de cobrança e obter acesso premium sem custo, falsificar desempenho e rankings através de registros de estudo fraudulentos, e acessar dados de assinatura de outros usuários. O impacto financeiro e de integridade é imediato e pode ser explorado sem conhecimento técnico avançado.

As medidas prioritárias são: implementar políticas RLS restritivas no Supabase para bloquear alterações diretas em campos de assinatura e plano; sincronizar estado de assinatura exclusivamente via webhooks validados do Stripe no backend; implementar validação server-side completa para todos os campos de study_records com limites plausíveis de valores; adicionar rate limiting no endpoint de autenticação com bloqueio progressivo após tentativas falhas; habilitar e promover uso de 2FA; tratar a vulnerabilidade de bypass de premium como incidente (auditoria de user_plans, reversão de ativações indevidas); e realizar revisão completa das políticas de RLS de todas as tabelas do projeto Supabase.

Possuem exploits públicos disponíveis e/ou não necessitam de muito esforço ou conhecimento técnico para sua exploração.

Vulnerabilidade de Prioridade Crítica As vulnerabilidades nesta categoria podem levar a um impacto significativo na confidencialidade, integridade e/ou disponibilidade de sistemas e dados organizacionais se não forem tratadas imediatamente.

Vulnerabilidade de Alta Prioridade As vulnerabilidades nesta categoria requerem atenção imediata e um plano de ação robusto.

Vulnerabilidade de Média Prioridade As vulnerabilidades nesta categoria são menos urgentes, mas, em algumas circunstâncias, ainda podem representar uma ameaça ou consequência séria.

Vulnerabilidade de Baixa Prioridade As vulnerabilidades nesta categoria não são uma ameaça iminente, mas devem ser mitigadas para evitar problemas a longo prazo.
