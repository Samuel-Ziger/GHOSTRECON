// Google hacking → nova aba

Escreve a query com operadores (vê a referência). O URL é montado e o Google abre noutra aba com os resultados — igual ao fluxo dos dorks automáticos, mas manual.

Prefixar site: <target>
Prévia: —
Operadores de pesquisa (referência)
intitle:
Restringe resultados com palavras-chave específicas no título.
Ex.: intitle:"Index of" password
inurl:
Filtra URLs que contenham uma string na path/query.
Ex.: inurl:admin
filetype:
Limita a ficheiros de um tipo (extensão).
Ex.: filetype:pdf
intext:
Procura texto no conteúdo indexado (corpo da página).
Ex.: intext:password
site:
Restringe a um site ou domínio.
Ex.: site:wikipedia.org
related:
Sites relacionados ao URL/domínio indicado.
Ex.: related:wikipedia.org
cache:
Versão em cache do Google para um URL (quando disponível).
Ex.: cache:wikipedia.org
"…"
Correspondência exata à frase entre aspas.
Ex.: "Wikipedia"
+
Termo obrigatório (sinónimo histórico; hoje o espaço já implica AND).
Ex.: jaguar +car
-
Exclui termo dos resultados.
Ex.: jaguar speed -car
OR
Combina alternativas (maiúsculas ajudam o Google a interpretar).
Ex.: jaguar OR car
imagesize:
Em pesquisa de imagens: filtra por dimensão.
Ex.: imagesize:320x320
@
Associado a handles em redes sociais (contexto Google).
Ex.: @wikipedia
#
Hashtags em contextos onde o Google indexa.
Ex.: #wiki
$
Sugestão de preço em pesquisas comerciais.
Ex.: tênis $390
..
Intervalo numérico (ex.: preços).
Ex.: tênis $120..$540

AJUDA — ACHADOS

Os separadores filtram os cartões pela prioridade ou pelo tipo. A barra de números em baixo resume contagens enviadas pelo servidor durante o recon. Com fila de alvos, esses contadores somam todos os alvos da mesma corrida. Alguns tipos de cartão só aparecem com modo Kali ou módulos activos.

ALL
Todos os cartões da sessão actual, com ordenação por tipo (subdomínios, endpoints, parâmetros, depois scans e heurísticas).
🔴 HIGH
Apenas achados com prioridade alta (heurística de risco ou confirmações mais fortes).
🟡 MED
Prioridade média — candidatos a investigar sem ser o topo da fila.
🟢 LOW
Prioridade baixa — contexto, intel passiva ou ruído provável.
SUBS
Subdomínios com DNS (A/AAAA) que resolvem: crt.sh, VirusTotal, subfinder/amass (Kali), etc., fundidos no mesmo fluxo.
HOSTS
Subdomínios em que o subfinder (Kali) contribuiu para a descoberta; o texto do cartão inclui tool=subfinder no meta.
ENDPOINTS
URLs ou caminhos (Wayback, Common Crawl, sitemap/robots, superfície HTML, Google CSE, JS, etc.).
PARAMS
Nomes de parâmetros de query extraídos do corpus de URLs (e descoberta activa quando aplicável).
SECRETS
Padrões que parecem segredos (ficheiros JS, GitHub Code Search) — tratar como suspeitos até validares.
NMAP
Saída do nmap no modo Kali (serviços e versões na amostra analisada).
EXPLOIT.DB
Referências do searchsploit associadas a produto/versão detectados (modo Kali).
NUCLEI
Resultados do Nuclei (incluindo passagens com tags XSS/SQLi quando há candidatos no corpus).
WPSCAN
Enumeração WPScan (WordPress) só quando o recon passivo já indicou WordPress e a ferramenta existe no Kali. Por defeito exige WPSCAN_API_TOKEN (WPVulnDB); GHOSTRECON_WPSCAN_REQUIRE_API=0 permite scan só com enumeração. Os eventos na consola usam o prefixo [WPScan].
DALFOX
Resultados do Dalfox (XSS) no modo Kali, quando o pipeline activa essa fase.
XSS_VIBES
Após a fase XSS do modo Kali, o servidor pode executar xss_vibes (pasta Xss/xss_vibes) e adicionar os hits como findings xss com meta scanner=xss_vibes.
XSS
Achados classificados como XSS (ex.: templates Nuclei ou fase Verify).
SQLI
Achados classificados como injecção SQL (Nuclei, Verify, etc.).
LFI
Inclusão local de ficheiros / path traversal levantados na verificação com evidência.
REDIRECT
Open redirect (fase Verify ou ferramentas que emitam esse tipo).
IDOR
Inseguranças de controlo de acesso / IDOR reportadas pelo pipeline de verificação.
TAKEOVER
Candidatos ou confirmações de subdomain takeover (CNAME, páginas de fornecedor, etc.).
SECRET-CHECK
Resultado da validação de segredos: live, provável ou morto (tipo secret_validation).
WHOIS
Texto de whois sobre o domínio ou subdomínios (modo Kali).
SEC
Problemas em cabeçalhos HTTP de segurança (HSTS, CSP, X-Frame-Options, cookies, etc.).
TLS
Certificado TLS: validade, SAN, emissor — alertas se a expiração estiver próxima.
RDAP
Metadados de registo de domínio via RDAP (estado, nameservers, eventos).

BARRA INFERIOR — CONTADORES

Valores actualizados em tempo real com o stream do pipeline. Não são o total de linhas na lista (há tipos sem contador próprio, ex.: tech, intel).

Subdomínios
Número de achados do tipo subdomain (hosts com A/AAAA após enumeração).
Endpoints
Achados endpoint e rotas extraídas de JS (js) que o pipeline contabiliza como superfície URL.
Parâmetros
Quantidade de achados param (nomes de query vistos no corpus ou descobertos activamente).
Secrets
Achados secret (padrões sensíveis em JS, GitHub, etc.) — muitos são para triagem manual.
Dorks
Quantidade de queries Google Hacking geradas como cartões dork neste run.
Alto Risco
Total de cartões com prioridade high após a última actualização de stats (inclui vários tipos, não só subs/endpoints).

// API · specs · GraphQL

Liga Wayback e/ou Common Crawl (ou outras fontes de URLs) para o corpus incluir caminhos úteis. O GraphQL só corre se existir URL com graphql nesse corpus (máx. 10). OpenAPI tenta paths típicos (/openapi.json, /swagger.json, …) em cada origem HTTPS viva.

OpenAPI / Swagger leak
GET a paths comuns; se a resposta for JSON de spec, extrai paths e parâmetros como achados.
GraphQL — introspecção mínima
Um POST com query mínima de introspecção; se o servidor responder, gera intel — rever manualmente e respeitar regras do programa.
Micro-exploit XSS (pós-verify)
Depois do verify, segunda variante de payload em XSS probable (pedidos GET activos).
Stealth (jitter + UA rotativa)
Atraso aleatório entre alguns pedidos e rotação de User-Agent; também pode activar-se com GHOSTRECON_STEALTH=1 no .env.

// Shannon (white-box local)

O que faz: o Shannon (Keygraph) corre em Docker uma análise white-box sobre código-fonte — clona repositórios Git (do programa de bounty ou sugeridos pelo recon), executa o workflow (Temporal) e gera um relatório de avaliação de segurança no workspace. Complementa o recon OSINT/web; não substitui dorks nem scans de superfície.

O que recebes no Ghost: por repositório processado aparece um cartão intel — em sucesso, com título tipo «Shannon white-box: org/repo», meta com workspace, caminho do ficheiro Markdown do relatório e um excerto do texto; em falha, cartão com fase/erro e dica no log. Podes abrir a Web UI do Temporal (quando o pipeline mostra o link) para acompanhar o workflow. O relatório completo fica no disco do workspace Shannon (ficheiro longo tipo assessment).

Módulo shannon_whitebox em modules. Código em IAs/shannon (não versionado no Git do Ghost): pnpm build, ./shannon build, imagem shannon-worker. A linha Shannon: … na sidebar resume Docker / pasta / CLI / imagem.

Módulo Shannon white-box
Envia shannon_whitebox em modules. O servidor pode recusar o POST /api/recon/stream se as dependências falharem (excepto «Omitir verificação»). No browser, o pré-check opcional volta a consultar /api/capabilities e pode sugerir docker pull.
Repos GitHub (manual)
Textarea com URLs de programa (https://github.com/…), owner/repo ou uma linha por entrada — junta-se à pesquisa API e entra no clone local com prioridade. Útil em bug bounty quando queres só esses repos (com Shannon sozinho, até 10 válidos).
Temporal / monitor
Quando o fluxo expõe URL da Web UI do Temporal, o log pode abrir o monitor — útil para acompanhar o workflow Shannon.

// PentestGPT (pós-recon HTTP)

O que faz: depois de todo o recon e da priorização (score), o servidor envia por POST um JSON com o resumo do run (alvo, estatísticas, amostra de findings, correlação) para o URL que configurares — em geral uma ponte que chama um LLM (ex.: OpenRouter) para triagem e texto orientado a bug bounty (o serviço não «ataca» o alvo; interpreta dados já recolhidos). Isto é independente do agente TUI/Python do clone GreyDGL em IAs/PentestGPT (Docker + make connect); esse repositório é o fluxo upstream «oficial», enquanto o módulo da UI usa só HTTP.

O que recebes no Ghost: o pipeline tenta mapear a resposta para novos achados (prioridade/tipo conforme o JSON). Se vier sobretudo texto agregado, podes ver um cartão intel «PentestGPT (resumo)» com esse texto; tudo entra na lista de findings e no gravado do run (SQLite / cloud) como os outros itens. O passo pentestgpt na barra do pipeline mostra quando corre ou salta.

Validação PentestGPT (HTTP)
Define GHOSTRECON_PENTESTGPT_URL no .env (ex.: http://127.0.0.1:8765/validate) e corre a ponte node server/scripts/pentestgpt-ghost-bridge.mjs (ou npm run pentestgpt-bridge) com OPENROUTER_API_KEY. Sem URL (nem override no campo da sidebar), o passo é ignorado. GHOSTRECON_PENTESTGPT_ENABLED=0 força skip.
URL POST opcional · Lembrar · Testar ponte
O campo «URL POST» envia pentestgptUrl no stream e sobrepõe o .env nessa corrida. «Lembrar» grava em localStorage. «Testar ponte» chama o servidor (POST /api/pentestgpt-ping), que faz GET …/health na mesma origem — evita bloqueios de CORS no browser.
Linha PentestGPT: … POST .env
A primeira parte descreve a árvore upstream em disco (pyproject, Python 3.12+, uv, Docker). A parte POST .env indica se GHOSTRECON_PENTESTGPT_URL está definida no servidor (não reflecte só o campo opcional).

// IA · relatórios (Gemini + Claude)

As caixas Confirmo envio para APIs externas e LM Studio no final (pré-check obrigatório) não usam a classe mod — não aparecem em modules[]; enviam autoAiReports e aiProviderMode no corpo do recon. Com confirmação: ao terminar cada recon (ou cada alvo na fila), o servidor envia o JSON do run às APIs configuradas e grava relatorio + proximos_passos em Markdown no disco; os próximos passos também são ecoados no log do terminal. Sem confirmação não há chamadas cloud. GHOSTRECON_AI_AUTO=0 no .env desliga a geração automática no servidor mesmo com a caixa marcada. Chaves: GEMINI_API_KEY ou GOOGLE_AI_API_KEY; segundo relatório via OPENROUTER_API_KEY (recomendado) ou ANTHROPIC_API_KEY se não houver OpenRouter. Com só LM Studio, a ordem no servidor mantém-se (Gemini → OpenRouter → Claude); LM Studio só entra se todas as anteriores falharem — antes do recon corre GET /api/ai/lmstudio-check. Cumpre sempre o programa de bounty.
